# Service Worker

The package handles the **PHP side** of service worker integration: headers, registration script,
and serving the SW file. The actual service worker code is generated by
[vite-plugin-pwa](https://vite-pwa-org.netlify.app/) / Workbox at build time.

---

## Strategies

### `generateSW` (default)

Workbox generates a complete service worker. No custom SW file is needed. Configure workbox options
through the `laravelPwa()` Vite plugin.

```env
PWA_SW_STRATEGY=generateSW
```

### `injectManifest`

You write your own service worker and Workbox injects the precache manifest into it. Use this when
you need custom fetch handling, push notifications, or background sync.

```env
PWA_SW_STRATEGY=injectManifest
```

Publish the stub:

```bash
ddev artisan vendor:publish --tag=laravel-pwa-stubs
```

Edit `resources/js/sw-custom.js` (the stub), then reference it in `vite.config.js`:

```js
laravelPwa({
    strategies: 'injectManifest',
    srcDir: 'resources/js',
    filename: 'sw-custom.js',
    manifest: false,
})
```

## Registration

### Blade directive (recommended)

```blade
{{-- Place before </body> --}}
@pwaRegisterSW
```

This renders an inline `<script>` that registers the SW on page load. The script respects:

- `pwa.service_worker.auto_register` — skip registration if `false`
- `pwa.service_worker.register_type` — `autoUpdate` posts `SKIP_WAITING` to the new worker;
  `prompt` leaves the decision to the user
- `pwa.service_worker.dev_enabled` — registers in `local` environment only when `true`

### Programmatic registration (JS companion)

If `auto_register` is `false` you can control timing from JavaScript:

```ts
import { setupPwa } from '@slfomin/pwa-laravel';

setupPwa({
    onNeedRefresh() {
        // Show your custom "update available" UI
    },
    onOfflineReady() {
        console.log('App ready for offline use');
    },
});
```

## Headers

The `PwaHeaders` middleware sets correct HTTP headers on SW and manifest responses. It is applied
automatically to the manifest route. For the SW file (served statically or via a custom route) add
the middleware to your web server or to the route:

```php
// routes/web.php
Route::get('/sw.js', fn () => response()->file(public_path('sw.js')))
    ->middleware('pwa.headers');
```

Default headers set by the middleware:

| Resource | Header | Value |
|---|---|---|
| Service Worker | `Content-Type` | `application/javascript; charset=utf-8` |
| Service Worker | `Cache-Control` | `no-cache, no-store, must-revalidate` |
| Service Worker | `Service-Worker-Allowed` | `/` |
| Manifest | `Content-Type` | `application/manifest+json` |
| Manifest | `Cache-Control` | `public, max-age=3600` |

Override headers in `config/pwa.php` under the `headers` key.

## ViteManifestBridge

`ViteManifestBridge` reads `public/build/manifest.json` (the Vite asset manifest, **not** the PWA
manifest) and resolves hashed asset filenames:

```php
use SlFomin\PwaLaravel\ServiceWorker\ViteManifestBridge;

$bridge = app(ViteManifestBridge::class);
$bridge->exists();              // bool — has Vite been built yet?
$bridge->asset('resources/js/app.js'); // '/build/app.abc123.js'
```

## WorkerManager

`WorkerManager` exposes the SW configuration and generates the registration script:

```php
use SlFomin\PwaLaravel\ServiceWorker\WorkerManager;

$worker = app(WorkerManager::class);

$worker->registrationUrl();     // '/sw.js'
$worker->scope();               // '/'
$worker->registerType();        // 'autoUpdate'
$worker->isAutoRegister();      // true
$worker->shouldRegisterInDev(); // false
$worker->isAvailable();         // true once Vite has been built
$worker->registrationScript();  // raw JS for the <script> tag
```

## Dev mode

By default the service worker is **not** registered in the `local` environment. This prevents
Vite HMR from being blocked by an outdated cache.

Enable it only when you specifically need to test offline behaviour locally:

```env
PWA_SW_DEV=true
```

## Runtime caching patterns

Configure `runtimeCaching` inside `laravelPwa()` in `vite.config.ts`.

### API: NetworkFirst

Tries the network first; falls back to cache on failure. Suitable for JSON APIs where
fresh data is preferred but stale data is acceptable when offline.

```ts
{
    urlPattern: /^\/api\//,
    handler: 'NetworkFirst',
    options: {
        cacheName: 'api-v1',
        networkTimeoutSeconds: 5,
        expiration: {
            maxEntries: 100,
            maxAgeSeconds: 60 * 60 * 24,       // 1 day
        },
        cacheableResponse: { statuses: [0, 200] },
    },
},
```

### Static assets: CacheFirst

Assets with content-hashes in their filename never change — serve them instantly from
cache. Only hits the network on a cache miss (first visit or after a deploy).

```ts
{
    urlPattern: /\/build\/assets\/.+\.[0-9a-f]{8}\.(js|css)$/i,
    handler: 'CacheFirst',
    options: {
        cacheName: 'static-assets-v1',
        expiration: {
            maxEntries: 200,
            maxAgeSeconds: 60 * 60 * 24 * 365,  // 1 year
        },
    },
},
```

### Images: StaleWhileRevalidate

Returns cached image immediately, then revalidates in the background. Good balance of
speed and freshness for non-critical images.

```ts
{
    urlPattern: /\.(?:png|jpg|jpeg|svg|gif|webp|avif)$/i,
    handler: 'StaleWhileRevalidate',
    options: {
        cacheName: 'images-v1',
        expiration: {
            maxEntries: 60,
            maxAgeSeconds: 60 * 60 * 24 * 30,   // 30 days
        },
    },
},
```

### Fonts: CacheFirst

Fonts are large and rarely change. Cache them for a long time.

```ts
{
    urlPattern: /^https:\/\/fonts\.(googleapis|gstatic)\.com\/.*/i,
    handler: 'CacheFirst',
    options: {
        cacheName: 'google-fonts-v1',
        expiration: {
            maxEntries: 30,
            maxAgeSeconds: 60 * 60 * 24 * 365,  // 1 year
        },
        cacheableResponse: { statuses: [0, 200] },
    },
},
```

### Cache invalidation on deploy

Precached assets (files matched by `globPatterns`) are invalidated automatically by
Workbox on each deploy — their revision hash changes. Runtime-cached assets with
content hashes (e.g. `/build/assets/app.abc123.js`) are effectively self-invalidating.

For runtime-cached URLs without hashes (e.g. `/api/config`), bump the `cacheName`
(e.g. `api-v2`) when you need to force a cache clear. Workbox cleans up old caches
via `cleanupOutdatedCaches: true` (enabled by default in this package).
